## 확장된 클라우드 및 서버 지원

Evertag는 이제 더 광범위한 클라우드 및 자가 호스팅 저장소에 네이티브로 연결됩니다. 어디에 있든 파일에서 ID3, MP4, FLAC, OGG, APE 태그를 편집할 수 있습니다.

### 프라이버시 중심 클라우드: Internxt와 Proton Drive

엔드 투 엔드 암호화와 제로 지식 저장에 신경을 쓴다면, 프라이버시 우선 클라우드 가운데 가장 인정받는 두 곳이 이제 Evertag에 네이티브로 들어왔습니다.

- **Internxt** — 오픈 소스, 포스트 양자 암호화, GDPR 준수의 스페인 클라우드. 무료 티어 제공.
- **Proton Drive** — Proton Mail과 Proton VPN 제작자가 만든 엔드 투 엔드 암호화 저장소(스위스 본사). 무료 티어 제공, 더 큰 라이브러리를 위한 유료 요금제도 있습니다.

이제 두 서비스에 저장된 오디오 파일에서 메타데이터를 직접 편집할 수 있습니다 — 파일은 전송 중에도 암호화된 상태이며, 새 태그만 다시 기록됩니다.

### 자가 호스팅 솔루션: QNAP, Nextcloud, Amazon S3

자체 인프라를 운영하는 사용자라면:

- **QNAP** — QNAP NAS 장치에 네이티브 API 연결. 로컬 Wi-Fi 또는 원격 접속을 통해 QNAP의 파일에 있는 태그를 편집합니다.
- **Nextcloud** — 자체 호스팅 또는 관리형 Nextcloud 인스턴스 모두에 연결할 수 있습니다.
- **Amazon S3** — Evertag를 임의의 S3 버킷(또는 Backblaze B2, Wasabi, MinIO, Cloudflare R2와 같은 S3 호환 저장소)에 연결하고 그 자리에서 메타데이터를 편집하세요.

### 새로운 네트워크 프로토콜: FTP, SFTP, NFS

맞춤형 서버, 홈랩, 정교한 모바일 앱이 없는 일반 NAS를 사용하는 분들을 위해 Evertag 4.2는 세 가지 클래식한 프로토콜을 추가했습니다.

- **SFTP(SSH File Transfer Protocol)** — **자체 서버에서 안전하게 원격 태그 편집**할 때 가장 적합한 답입니다. SFTP는 SSH 위에서 동작하므로 인증과 오디오 데이터 등 전체 전송이 암호화됩니다. SSH 접근이 가능한 VPS, 전용 서버, 집의 Linux 머신이 있다면 다른 어떤 것도 노출하지 않은 채 원격 파일의 태그를 편집할 수 있습니다. 비밀번호 및 키 기반 인증을 모두 지원합니다.
- **FTP** — 오랫동안 자리 잡은 파일 전송 표준입니다. FTP를 노출하지만 네이티브 API가 없는 더 오래된 가정용 NAS에 유용합니다.
- **NFS(Network File System)** — Linux와 대부분의 NAS에서 사실상 표준이 된 공유 프로토콜입니다. 같은 하드웨어에서 SMB보다 오버헤드가 적습니다.

> **팁:** 공개 인터넷에서 원격 편집을 원한다면 SFTP를 사용하세요. FTP와 NFS는 로컬 네트워크 안에서 사용하는 것이 좋습니다 — VPN으로 감싸지 않는 한 공개 인터넷에 노출하지 마세요.

## Wi-Fi Drive 업그레이드

[**Wi-Fi Drive**](/docs/howto/how-to-transfer-files-wirelessly-from-a-computer-to-an-iphone-using-wifi-drive/)는 **iTunes, 케이블, 클라우드 계정 없이 컴퓨터와 iPhone 또는 iPad 사이에서 오디오 파일을 무선 전송**하기 위한 Evertag 내장 기능입니다. 두 기기는 같은 Wi-Fi 네트워크에 연결되어 있어야 합니다.

Evertag 4.2에서 Wi-Fi Drive는 다음을 갖춥니다.

- **새로워진 모던 인터페이스** — 깔끔하고, 한눈에 읽기 쉽고, Liquid Glass에 맞춰 업데이트.
- **다중 선택 모드** — 여러 파일이나 폴더를 선택해 일괄 처리합니다.
- **더 똑똑해진 업로드 큐** — 진행 상황 피드백이 더 명확하고 네트워크 변동에 더 강합니다.
- **속도와 전반적 안정성 향상** — 큰 라이브러리에서 더 빠른 전송 속도.

타사 서비스 없이 컴퓨터에서 휴대폰으로 오디오 파일 묶음을 옮기고 태그를 편집한 뒤 다시 보내는 가장 빠른 방법입니다.

## 태그 편집기 설정: 자세히 살펴보기

대부분의 사용자가 절대 읽지 않는 부분이지만 — 태그가 어디서나 작동할지, 일부 앱에서만 작동할지를 결정하는 부분입니다. Evertag를 열고 **오디오 태그 편집기 설정** 섹션으로 이동하세요. 각 옵션이 실제로 무엇을 하는지, 무엇을 달성하려는지에 따라 어느 옵션을 선택해야 하는지 정리했습니다.

### 앨범 커버 스케일링

오디오 파일에 앨범 아트를 저장할 때 Evertag는 임베드 전에 이미지를 축소할 수 있습니다. 사용 가능한 옵션은 다음과 같습니다.

- **소** — 파일 크기에 미치는 영향이 가장 작고 이미지 품질도 낮음.
- **중** — 대부분의 사용자에게 균형 잡힌 선택.
- **대** — 고품질, 큰 화면 플레이어와 CarPlay에 적합.
- **특대** — 매우 고품질, 파일 크기가 눈에 띄게 증가.
- **원본(비활성화)** — 원본 해상도로 임베드. **스케일링 없음.**

**이게 왜 중요한가:**

- **고품질 = 더 큰 파일.** 3,000 × 3,000픽셀 커버는 트랙당 수 MB를 추가할 수 있습니다. 앨범 전체로 곱하면 디스크 비용이 빠르게 누적됩니다.
- **일부 플레이어는 매우 큰 임베디드 이미지를 잘 처리하지 못합니다.** 오래된 기기, 일부 차량용 헤드 유닛, 일부 레거시 데스크톱 플레이어는 ~1,500 픽셀을 넘는 커버에서 멈추거나 표시를 거부할 수 있습니다.
- **대부분의 클라우드 및 스트리밍 워크플로우에서는** **중** 또는 **대**가 적절한 선택입니다. **원본**은 아카이브 품질이 필요하거나 신뢰하는 플레이어용 파일을 준비할 때만 사용하세요.

> **원본** 크기 옵션은 Evertag 프리미엄 개인화 업그레이드의 일부입니다. 표준 크기(소/중/대/특대)는 무료입니다.

### 태그 저장 옵션: ID3v2.4 vs ID3v2.3

호환성에서 가장 중요한 단일 설정입니다. ID3v2는 MP3 파일 내에서 사용되는 메타데이터 형식입니다. 두 가지 널리 사용되는 버전이 있고, 미묘하지만 중요한 차이가 있습니다.

#### ID3v2.4

- 더 새롭고 **UTF-8 텍스트 인코딩**을 지원합니다 — 비라틴 문자(중국어, 러시아어, 일본어, 아랍어, 히브리어 등)를 적절히 처리합니다.
- 더 많은 프레임 유형(상대 볼륨, 이퀄라이저 프리셋 등).
- 이를 지원하는 **현대적인 플레이어에 권장**됩니다.

#### ID3v2.3

- 더 오래되었지만 **가장 보편적으로 지원되는 ID3 버전**입니다.
- UTF-8을 직접 지원하지 않습니다(유니코드 텍스트에는 UTF-16 사용).
- 레거시 플레이어, 차량용 스테레오, 일부 데스크톱 앱과 **최대 호환성이 필요할 때 권장**됩니다.

#### Evertag에서 ID3v2.4를 켜야 할 때

- Evermusic, Plex, Jellyfin, Navidrome, foobar2000, VLC, Apple Music(현재 버전) 또는 현대적인 Android 플레이어 같은 **현대적인 오디오 플레이어**를 사용합니다. ✅ **ID3v2.4를 켭니다.**
- 라이브러리에 **비라틴 문자**(중국어, 한국어, 일본어, 러시아어, 아랍어, 그리스어, 히브리어)가 있습니다. ✅ **ID3v2.4를 켭니다** — UTF-8이 이를 훨씬 깨끗하게 처리합니다.

#### Evertag에서 ID3v2.4를 꺼야 할 때(대신 ID3v2.3 사용)

- v2.4 태그를 읽지 않는 **더 오래된 차량용 스테레오 또는 인대시 유닛**용 파일을 준비합니다.
- 편집 후 일부 앱에서 **깨진 텍스트나 누락된 태그**가 보입니다 — 그곳에서 v2.4가 지원되지 않는다는 강한 신호입니다. v2.3으로 되돌리세요.
- **레거시 데스크톱 플레이어**나 더 오래된 휴대용 플레이어(초기 iPod, 2000~2010년대의 일부 MP3 플레이어)를 대상으로 합니다.

> **경험적 규칙:** v2.4로 모든 곳에서 태그가 올바르게 표시된다면 그대로 두세요. 중요한 플레이어 하나라도 잘못된 문자, 빈칸을 보여주거나 태그를 읽지 못한다면 v2.4를 끄고 다시 저장하세요.

#### 태그 중복

**태그 중복**을 활성화하면 Evertag는 공통 메타데이터 필드(제목, 아티스트, 앨범 등)를 파일의 **ID3v1과 ID3v2 두 섹션에 모두** 작성합니다. 이는 ID3v1만 읽는 매우 오래된 플레이어(파일 끝의 원래 128바이트 태그)와의 호환성을 향상시킵니다.

- **대부분의 사용자에게는 필요하지 않습니다.** 현대 플레이어는 ID3v2를 먼저 읽습니다.
- 빈티지 하드웨어나 ID3v2를 무시하는 특정 소프트웨어를 다루는 경우 **에만 활성화**하세요.

### 온라인 파일 업데이트: Evertag가 클라우드 편집을 처리하는 방식

연결된 클라우드(Google Drive, Dropbox, OneDrive, iCloud, Internxt, Proton Drive, QNAP, Nextcloud, Amazon S3, SFTP 등)에 저장된 파일의 태그를 편집할 때, Evertag는 수정된 파일을 다시 업로드해야 합니다. 어떻게 처리할지는 사용자가 선택합니다.

- **확인 메시지 표시** *(기본값, 권장)* — Evertag가 업로드 전에 묻습니다. 신중한 사용자와 공유 라이브러리에 가장 적합합니다.
- **파일 메타데이터 자동 업데이트** — 각 편집 후 조용히 업로드합니다. 빠르고 안정적인 연결로 자주 편집하는 단독 사용자에게 가장 적합합니다.
- **파일 메타데이터 업데이트 안 함** — Evertag는 로컬 사본만 편집합니다. 클라우드에 반영하지 않고 태그 변경을 미리 볼 때 유용합니다.

### 온라인 파일 편집: 로컬 파일 정리

원격 파일을 편집하려면 Evertag가 먼저 기기로 다운로드합니다. 편집 후, 그 로컬 사본을 어떻게 할지 선택합니다.

- **다운로드한 파일 항상 삭제** — 편집 후 Evertag가 임시 파일을 제거합니다. 저장 공간이 부족하거나 다른 사람의 파일에서 작업할 때 **권장**됩니다.
- **다운로드한 파일 삭제 안 함** — 편집된 파일을 기기에 유지합니다. 오프라인 사본과 업데이트된 클라우드 사본을 모두 갖고 싶을 때 유용합니다.

### 메인 화면의 버튼

Evertag의 태그 편집기 홈 화면은 개별 작업의 버튼을 표시하거나 숨길 수 있습니다. UI를 집중되게 유지하려면 실제로 사용하는 버튼만 켜세요.

- **오디오 태그 자동 검색** — 파일의 오디오 지문을 기반으로 누락된 메타데이터를 온라인에서 찾습니다.
- **오디오 태그 수동 검색** — 자동 검색이 실패할 때 제목/아티스트로 검색합니다.
- **앨범 아트워크 검색** — 고품질 커버를 찾아 임베드합니다.
- **앨범 아트워크 저장** — 임베디드 커버를 사진 라이브러리나 파일로 내보냅니다.
- **인코딩 정규화** — 오래된 인코딩으로 인한 깨진 비라틴 텍스트를 수정합니다(레거시 소프트웨어로 리핑한 키릴, 중국어, 일본어 트랙에 매우 유용함).
- **오디오 태그 삭제** — 파일에서 모든 태그를 제거합니다. 새로 태깅하기 전에 유용합니다.

### 확장 태그 표시

이를 토글하면 기본 제목/아티스트/앨범/연도를 넘어선 전체 메타데이터 필드 — BPM, 지휘자, 원본 아티스트, 분위기, 저작권, 인코더, 댓글, 가사 등 — 가 표시됩니다. 파워 사용자 기능이며, 대부분의 일반 사용자에게는 필요하지 않습니다.

### 파일 동시 편집

활성화되면 Evertag는 **선택한 여러 파일에 걸쳐** 메타데이터를 편집할 수 있게 합니다 — 한 번의 작업으로 전체 앨범에 동일한 album artist, 연도, 장르를 설정합니다. 크고 정리되지 않은 라이브러리를 정리하는 가장 빠른 방법입니다.

## Spotify, Apple Music, 스트리밍 플랫폼용 태그 편집

자주 받는 질문: "Evertag에서 태그를 편집하고 파일을 업로드했는데 스트리밍 서비스가 잘못된 메타데이터를 보여줍니다. 어떻게 된 거죠?"

짧은 답: **스트리밍 서비스는 항상 로컬 태그를 존중하지는 않습니다.** Apple Music과 Spotify에는 자체 내부 데이터베이스가 있습니다 — 트랙을 인식하면 표시되는 메타데이터를 자체 데이터로 덮어씁니다. 그러나 **업로드된 파일**, 로컬 파일(Apple Music의 「라이브러리」 기능, Spotify Local Files), **유통사를 통한 스트리밍 플랫폼 업로드**에서는 태그가 절대적으로 중요합니다. 각 시나리오에 맞춰 Evertag를 설정하는 방법은 다음과 같습니다.

### Apple Music용 파일 준비(Cloud Music Library / iTunes Match)

- **ID3v2.4: ON** — Apple Music은 UTF-8을 올바르게 처리합니다.
- **앨범 커버: 대** — Apple 앱은 큰 아트워크를 잘 렌더링합니다. 원본은 과합니다.
- **태그 중복: OFF** — 필요하지 않습니다.
- **Album Artist**가 올바르게 채워져 있는지 확인하세요 — Apple Music이 그룹화에 사용합니다.

### Spotify Local Files용 파일 준비

Spotify Local Files는 잘 태깅된 파일만 표시합니다. Spotify가 읽는 태그는 제한적입니다.

- 대부분의 경우 **ID3v2.4: ON**. 편집 후 트랙이 Spotify에서 올바르게 표시되지 않는다면 **ID3v2.4: OFF**(즉, ID3v2.3)로 저장해 보세요 — Spotify의 파서는 역사적으로 v2.4에 보수적이었습니다.
- **앨범 커버: 중 또는 대** — Spotify는 어차피 아트워크를 축소합니다.
- 최소한 **제목**, **아티스트**, **앨범**, **트랙 번호**를 채우세요.

### 유통사 업로드용 파일 준비(DistroKid, TuneCore, CD Baby 등)

자신의 작품을 스트리밍 플랫폼에 업로드하는 아티스트라면 유통사가 보통 태그를 읽지만 자체 UI에서도 메타데이터를 요구합니다. 어쨌든:

- **ID3v2.3**이 더 안전한 기본값인 경우가 많습니다 — 많은 유통사 파이프라인은 수년 전에 구축되어 더 오래된 형식을 선호합니다.
- **대** 아트워크를 임베드하세요(대부분의 유통사는 ≥ 1,400 × 1,400픽셀 아트워크를 요구 — 가이드라인을 확인하세요).
- 확장 태그에 의존하지 마세요 — 유통사는 핵심 필드만 읽습니다.

### Plex, Jellyfin, Navidrome, Subsonic, Emby용 파일 준비

이러한 자가 호스팅 미디어 서버는 매우 관대합니다. v2.3과 v2.4 모두 깨끗하게 읽고 UTF-8을 잘 처리합니다.

- **ID3v2.4: ON**도 괜찮습니다.
- **앨범 커버: 대** 또는 **특대** — 이 서버들은 모바일 클라이언트와 CarPlay 화면에 아트워크를 제공하므로 품질이 중요합니다.
- **Album Artist**를 강력 권장합니다 — Plex/Jellyfin이 아티스트별로 앨범을 올바르게 그룹화하는 데 사용합니다.

### 차량용 스테레오 및 오래된 하드웨어용 파일 준비

- **ID3v2.4: OFF**(ID3v2.3 사용) — 더 오래된 헤드 유닛은 v2.4를 지원하지 않는 경우가 많습니다.
- **앨범 커버: 중** — 많은 차량용 디스플레이가 큰 임베디드 아트로 인해 멈춥니다.
- **태그 중복: ON** — 더 오래된 차량용 스테레오는 가끔 ID3v1 폴백만 읽습니다.
